# 1. 项目定位与建设目标

系统面向多水库管理场景，目标是把自动站数据、人工采样、标准规范、处置经验、污染源关系和报告流程统一到一个平台内。核心链路为：数据接入、实时展示、异常识别、RAG 检索、图谱溯源、Agent 生成建议、人工确认闭环。

| **设计项** | **实现要求**                                                                                   |
|------------|------------------------------------------------------------------------------------------------|
| 数据驱动   | 以自动监测站数据为主，人工采样和遥感数据作为补充；数据入库前完成量程、变化率、跨站一致性校验。 |
| 智能预警   | 同时支持规则阈值预警、趋势异常预警、蓝藻水华条件预警和设备健康诊断。                           |
| 知识增强   | 标准、案例、预案统一进入知识库，RAG 回答必须返回来源信息，禁止无依据生成。                     |
| 图谱溯源   | 以水库、河流、污染源、监测站、指标为实体，支持沿水系上游多跳追溯和污染源排序。                 |
| Agent 协同 | 通过 LangGraph 编排巡检、预警、应急、报告、调度工作流，关键处置节点保留人工确认。              |
| 业务闭环   | 预警从发现、推送、确认、处置、复盘到归档形成完整状态流转。                                     |

# 5. 核心模块技术设计

## 5.1 多源数据接入与治理

| **设计项** | **技术实现要求**                                                                                     |
|------------|------------------------------------------------------------------------------------------------------|
| 输入       | 自动监测站 RTU 数据、人工采样录入、遥感反演数据、模拟监测数据。                                      |
| 处理       | 统一字段映射；按站点、指标、时间进行去重；执行量程、变化率、跨站一致性校验；生成 quality_flag。      |
| 存储       | 明细数据写入 monitoring_record；最新数据写入 Redis；异常入库 alert_event；数据源记录在 data_source。 |
| 接口       | 提供批量导入、单条上报、最新数据、历史趋势、数据质量查询接口。                                       |
| 约束       | monitoring_record 必须建立 reservoir_id + record_time、indicator_id + record_time 联合索引。         |

## 5.2 实时监测看板

| **设计项** | **技术实现要求**                                                              |
|------------|-------------------------------------------------------------------------------|
| 页面       | 首页总览、水库列表、水库详情、指标趋势、多库对比、站点状态、异常列表。        |
| 数据       | 首屏从 MySQL 聚合读取，最新值优先读 Redis；趋势数据按时间窗口分页或聚合返回。 |
| 实时       | FastAPI WebSocket 推送最新监测值、站点在线状态、预警事件变更。                |
| 图表       | ECharts 展示折线图、柱状对比、仪表盘、异常标记和水质等级分布。                |
| 性能       | 按水库和指标维度缓存近实时数据；长时间序列查询使用后端聚合减少前端渲染压力。  |

## 5.3 智能预警中心

| **设计项** | **技术实现要求**                                                                  |
|------------|-----------------------------------------------------------------------------------|
| 预警类型   | 国标超标、阈值接近、趋势异常、蓝藻水华条件、设备疑似故障、综合事件。              |
| 规则引擎   | 基于 indicator 标准限值、水库目标等级、历史均值、连续异常次数、站点状态进行判定。 |
| AI 增强    | 预警触发后调用 RAG 检索预案，调用 Neo4j 获取污染源候选，由 LLM 生成处置建议。     |
| 状态流转   | new、confirmed、processing、resolved；所有状态变更记录处理人、时间和备注。        |
| 去重       | 同一水库、同一指标、同一时间窗口内预警合并；综合事件按流域和空间邻近规则合并。    |

## 5.4 RAG 智能问答

| **设计项** | **技术实现要求**                                                                              |
|------------|-----------------------------------------------------------------------------------------------|
| 知识类型   | 国家标准、地方条例、历史案例、应急预案、处置记录、巡检报告。                                  |
| 入库流程   | 上传文件、提取文本、清洗、按标题和段落切片、Embedding、写入 Chroma、记录 knowledge_document。 |
| 检索策略   | 向量相似度召回 + 元数据过滤；标准限值类问题优先查询 indicator 表，再补充文档来源。            |
| 生成要求   | 答案必须引用来源标题和片段位置；检索无结果时返回未找到，不允许编造标准。                      |
| 交互       | 前端使用 SSE 或流式接口展示回答，后端记录问题、召回文档、模型、耗时和用户反馈。               |

## 5.5 水质知识图谱

| **设计项** | **技术实现要求**                                                                     |
|------------|--------------------------------------------------------------------------------------|
| 实体       | 水库、河流、监测站、污染源、行政区、指标、事件、处置措施、供水管网节点。             |
| 关系       | 位于、上游、下游、流入、流出、影响、关联指标、发生事件、采用措施、供水到。           |
| 查询       | 从异常水库沿上游关系追溯指定距离或跳数，结合污染源类型、距离和历史事件生成候选排序。 |
| 可视化     | 后端输出节点、边、类型、权重和风险等级，前端用 ECharts 力导向图展示。                |
| 治理       | 新增实体时进行名称模糊匹配和坐标邻近去重；图谱变更记录版本，保证历史溯源可审计。     |

## 5.6 Agent 智能分析引擎

| 调度 Agent | 综合水质等级、蓄水量、预测来水、供水管网和下游需求生成调度建议。           |

# 6. 数据设计

## 6.1 MySQL 业务表

| **表名**           | **用途**           | **关键字段/关系**                                                                                 |
|--------------------|--------------------|---------------------------------------------------------------------------------------------------|
| reservoir          | 水库主数据         | name、code、location、longitude、latitude、capacity、water_grade、watershed、status。             |
| monitoring_station | 监测站点           | reservoir_id 关联水库；type 区分 auto、manual、sensing；last_data_time 判断离线。                 |
| indicator          | 水质指标和标准限值 | code、unit、category、standard_limit_i 至 standard_limit_v、is_core。                             |
| monitoring_record  | 监测明细           | reservoir_id、station_id、indicator_id、value、data_source、quality_flag、record_time；核心大表。 |
| role / user        | 角色和用户         | role.permissions 存 JSON 权限；user 关联 role；密码使用 bcrypt 哈希。                             |
| alert_event        | 预警事件           | reservoir_id、handler_id、alert_level、indicators JSON、source_desc、suggestion、status。         |
| inspection_report  | 巡检和统计报告     | report_type、summary、details JSON、alerts JSON、status、reviewer_id。                            |
| knowledge_document | 知识库文档元数据   | doc_type、file_path、content、metadata JSON、chunk_count、status。                                |
| disposal_plan      | 处置预案           | scenario、alert_indicators、steps、severity，用于预案检索和应急建议。                             |

## 6.2 Redis 设计

| **数据类型**   | **Key 设计**                                | **Value 内容**                                     | **过期策略**                           |
|----------------|---------------------------------------------|----------------------------------------------------|----------------------------------------|
| 最新监测值     | latest:{reservoir_id}:{indicator_code}      | value、unit、record_time、quality_flag、station_id | 设置 TTL，避免离线站点旧数据长期存在。 |
| 站点状态       | station:status:{station_id}                 | online/offline、last_data_time、health_score       | 短 TTL，由心跳或数据上报刷新。         |
| 预警去重窗口   | alert:dedup:{reservoir_id}:{indicator_code} | 最近预警事件 ID 和时间窗口                         | 按预警合并窗口过期。                   |
| Agent 会话状态 | agent:state:{task_id}                       | 工作流节点、输入摘要、输出结果、人工确认状态       | 任务完成后短期保留，归档写 MySQL。     |
| RAG 临时会话   | chat:{session_id}                           | 对话历史摘要、最近召回文档 ID                      | 按用户会话过期。                       |

## 6.3 Neo4j 图谱模型

| **类别** | **设计内容**                                                                                                             |
|----------|--------------------------------------------------------------------------------------------------------------------------|
| 节点标签 | Reservoir、River、MonitoringStation、PollutionSource、Indicator、AlertEvent、DisposalPlan、WaterSupplyNode、Area。       |
| 关系类型 | LOCATED_IN、UPSTREAM_OF、DOWNSTREAM_OF、FLOWS_TO、MONITORS、AFFECTS、RELATED_TO、HAS_EVENT、USES_PLAN、SUPPLIES_TO。     |
| 核心属性 | 节点保留 name、code、location、longitude、latitude、type、risk_level；关系保留 distance、weight、direction、updated_at。 |
| 溯源约束 | 查询深度建议不超过 5 跳；上游追溯按距离、污染源类型、历史事件次数和指标相关性综合排序。                                  |

## 6.4 Chroma 向量库设计

| **设计项** | **要求**                                                                                    |
|------------|---------------------------------------------------------------------------------------------|
| Collection | 按知识类型或项目统一为 water_quality_knowledge，metadata 区分 standard、case、sop、report。 |
| 切片策略   | 按标题、条款、段落切片；标准限值表格相关内容需人工校验或同步到 indicator 表。               |
| Metadata   | document_id、title、doc_type、source、section、chunk_index、created_at、tags。              |
| 索引维护   | 单次新增文档后可即时入库；累计新增超过阈值时触发批量重建；失败状态写回 knowledge_document。 |

# 7. 接口设计

| **接口组** | **核心接口**                                               | **实现说明**                                                 |
|------------|------------------------------------------------------------|--------------------------------------------------------------|
| 认证与用户 | 登录、刷新 Token、当前用户、用户 CRUD、角色 CRUD、权限查询 | JWT 鉴权；依赖注入当前用户；管理员才能维护角色权限。         |
| 水库与站点 | 水库列表/详情、站点列表/详情、指标配置、站点状态           | 基础配置接口由管理员维护；业务查询按角色过滤可见范围。       |
| 监测数据   | 最新数据、历史趋势、批量导入、人工采样录入、质量标记       | 写入前统一校验；趋势接口支持水库、指标、时间范围筛选。       |
| 实时推送   | WebSocket 订阅水库、预警、站点状态                         | 连接建立后校验 Token；服务端按事件类型推送增量数据。         |
| 预警中心   | 预警列表、详情、确认、处理、关闭、处置记录、推送重试       | 状态机由后端控制；禁止前端直接改写终态。                     |
| RAG 问答   | 文档上传、处理状态、知识检索、对话、反馈                   | 上传后异步切片向量化；对话接口返回答案和来源。               |
| 图谱分析   | 溯源查询、节点详情、邻居扩展、图谱导入、实体关系维护       | Neo4j 查询结果转换为前端图谱格式；查询设置深度和节点数上限。 |
| Agent 任务 | 触发巡检、触发应急分析、任务状态、人工确认、任务结果       | LangGraph 任务 ID 可追踪；人工确认节点必须支持超时处理。     |
| 报告中心   | 报告列表、生成、预览、审核、发布、导出                     | 报告详情结构化存储；生成任务异步执行。                       |

## 7.1 接口通用规范

- 所有管理类接口必须进行 JWT 鉴权和 RBAC 权限校验。

- 列表接口统一支持 page、page_size、sort、时间范围和关键字筛选。

- 写接口统一返回业务 ID、状态和可追踪 trace_id。

- 异步任务接口返回 task_id，任务状态通过轮询或 WebSocket 推送。

- 错误响应区分参数错误、权限错误、资源不存在、业务状态冲突、外部服务失败。

# 8. AI 工作流设计

## 8.1 RAG 问答流程

| **步骤**   | **实现**                                                                                |
|------------|-----------------------------------------------------------------------------------------|
| 问题解析   | 识别问题类型、指标名称、水库名称、时间范围和是否需要标准限值。                          |
| 结构化优先 | 涉及标准限值和监测值对比时，优先查询 indicator 和 monitoring_record，避免表格解析误差。 |
| 向量召回   | 将用户问题向量化，从 Chroma 召回标准、案例、预案等片段。                                |
| 重排与过滤 | 按相似度、文档类型、更新时间、用户角色权限过滤召回结果。                                |
| 生成       | Qwen-Max 基于结构化数据和召回片段生成回答，附来源标题和段落位置。                       |
| 记录       | 记录用户问题、召回结果、答案、模型、耗时、反馈，用于优化检索。                          |

## 8.2 预警与应急 Agent 流程

| **节点**     | **输入**                                   | **输出**                                  |
|--------------|--------------------------------------------|-------------------------------------------|
| 数据扫描     | 最新监测数据、站点状态、历史窗口           | 异常候选列表和质量判断。                  |
| 规则判定     | 指标限值、水库水质等级、趋势规则、设备规则 | 预警等级、触发原因、是否合并。            |
| 事件创建     | 预警候选、去重窗口、已有事件               | alert_event 记录和任务 ID。               |
| RAG 预案检索 | 预警指标、场景、等级                       | 相关标准、案例、处置预案片段。            |
| 图谱溯源     | 水库、指标、空间范围、追溯深度             | 污染源候选、路径、风险排序。              |
| 建议生成     | 预警事实、RAG 片段、图谱结果               | 原因分析、处置步骤、通知对象和风险说明。  |
| 人工确认     | 处置建议、负责人、超时策略                 | 确认、驳回、修改或升级。                  |
| 闭环归档     | 处置结果、复盘信息                         | 状态 resolved，沉淀为案例或预案优化素材。 |

## 8.3 报告生成流程

- 报告 Agent 根据 report_type 决定统计范围和模板结构。

- 数据层聚合水库达标率、主要超标因子、等级分布、同比/环比变化和设备状态。

- 图谱层补充异常事件的上游污染源、面源污染区域和影响链路。

- LLM 仅负责生成报告文字，不直接计算核心指标；所有数值由后端聚合产生。

- 报告生成后以 draft 状态保存，技术负责人审核后发布。

## 9.1 前端状态与权限

- Pinia 持久化用户信息、Token、权限集合、当前水库上下文和全局字典。

- 路由守卫按角色权限动态生成菜单；按钮级权限通过指令或组件封装控制。

- Axios 拦截器处理 Token 注入、401 重新登录、业务错误提示和下载响应。

- WebSocket 断线后指数退避重连；重连后重新订阅水库和预警频道。

# 10. 安全与权限设计

| **安全项** | **实现要求**                                                                   |
|------------|--------------------------------------------------------------------------------|
| 认证       | JWT Token 有效期 24 小时；前端做无感刷新或过期重登；敏感接口必须校验用户状态。 |
| 密码       | 使用 bcrypt 哈希存储；重置密码需要二次验证；禁止明文日志。                     |
| RBAC       | 角色权限以模块、页面、操作维度维护；后端接口必须二次校验，不依赖前端隐藏。     |
| CORS       | 生产环境限定前端域名，禁止使用通配配置。                                       |
| 密钥       | DeepSeek、Qwen、JWT Secret、数据库密码全部放后端环境变量，不下发前端。         |
| 文件上传   | 限制文件类型、大小和解析超时；知识库文档处理失败必须可追踪。                   |
| 审计       | 预警状态变更、报告审核、图谱编辑、知识库变更记录操作人和时间。                 |
