# 水库水质智慧监测平台 · 前端驱动功能模块与接口设计 v3（实际接口风格对齐版）

## 1. 文档说明

### 1.1 文档目标

本文档用于替换上一版《水库水质智慧监测平台 · 前端驱动功能模块与接口设计 v2》。

上一版已经完成了：

- 按前端 12 个页面重新梳理 M1 ~ M8 模块；
- 将原后端能力清单改造成页面驱动的联调口径；
- 识别前端页面所需的接口、WebSocket / SSE 能力与后置能力。

本版本在此基础上进一步调整：

- 以当前实际《API 接口文档》为准；
- 接口路径、命名方式、分页结构、响应结构均按实际实现风格编写；
- 对已经实现的接口明确标注“可直接联调”；
- 对前端页面需要但当前 API 文档尚未提供的接口明确标注“待补接口”；
- 避免把未实现能力写成已完成接口，减少前后端联调误差。

### 1.2 重要约定

| 项目 | 本版约定 |
|---|---|
| 基础 URL | `http://localhost:3443` |
| 认证方式 | Bearer Token，Header：`Authorization: Bearer <token>` |
| 统一响应 | `ApiResponse`：`{ code, message, data }` |
| 分页结构 | `data.lists` + `data.pagination` |
| 成功返回 | 查询接口返回对象 / 数组；增删改接口多返回 `data: true` |
| 已实现接口路径 | 以实际 API 文档为准，当前存在 `/api` 与 `/api/v1` 混用 |
| 新增接口建议 | 后续新增接口建议统一采用 `/api/v1` 前缀 |
| 前端适配原则 | 先适配实际已实现接口，再逐步补齐页面缺口 |

> 说明：此前曾约定 REST API 统一使用 `/api/v1`。但当前实际接口文档中，认证、用户、角色、水库、站点、指标、监测记录使用 `/api/...`，仪表盘和预警使用 `/api/v1/...`。本版为保证可联调，**不强行改写实际已实现路径**。如果后续必须统一 `/api/v1`，建议后端增加兼容路由别名，例如 `/api/v1/auth/login` 转发到 `/api/auth/login`。

## 2. 实际 API 通用规范

### 2.1 统一响应格式

所有业务接口返回 `ApiResponse`：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

错误响应：

```json
{
  "code": 1001,
  "message": "参数错误",
  "data": null
}
```

### 2.2 错误码

| 错误码 | 说明 |
|---:|---|
| 0 | 成功 |
| 1001 | 参数错误 |
| 1002 | 数据不存在 |
| 2001 | 未登录 / 用户不存在或已禁用 |
| 2002 | 令牌过期 |
| 2003 | 权限不足 |
| 2004 | 令牌无效 |
| 3001 | 文件格式不支持 |
| 3002 | 文件大小超过限制 |
| 4001 | 调用 AI 服务错误 |
| 5001 | 服务器内部错误 |
| 6001 | 密码错误 |
| 7001 | 资源已存在 |

### 2.3 分页返回结构

当前分页接口统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "lists": [],
    "pagination": {
      "page": 1,
      "page_size": 10,
      "total": 0,
      "total_pages": 0
    }
  }
}
```

前端统一适配建议：

| 前端期望字段 | 实际字段 |
|---|---|
| `items` | `data.lists` |
| `page` | `data.pagination.page` |
| `page_size` | `data.pagination.page_size` |
| `total` | `data.pagination.total` |
| `total_pages` | `data.pagination.total_pages` |

### 2.4 实际接口命名风格

当前实际接口不是标准 RESTful 风格，而是“资源路径 + 动作后缀”混合风格。

| 类型 | 实际风格示例 |
|---|---|
| 列表 | `/api/users/list`、`/api/reservoir/list`、`/api/stations/list` |
| 新增 | `/api/users/add`、`/api/reservoir/create`、`/api/stations/create` |
| 详情 | `/api/users/{id}`、`/api/reservoir/{id}` |
| 更新 | `/api/users/{id}`、`/api/roles/update` |
| 删除 | `/api/roles/{id}`、`/api/indicators/{id}` |
| 业务查询 | `/api/monitoring/last`、`/api/monitoring/trend` |
| 仪表盘 | `/api/v1/dashboard/overview`、`/api/v1/dashboard/reservoir-cards` |
| 预警 | `/api/v1/alerts`、`/api/v1/alerts/{id}` |

## 5. M8 系统管理模块

M8 覆盖登录、用户、角色、水库、站点、指标等系统基础能力，是当前实际接口覆盖最完整的模块。

### 5.1 登录页接口适配

#### 5.1.1 登录

| 项目 | 内容 |
|---|---|
| 方法 | POST |
| 实际路径 | `/api/auth/login` |
| 请求体 | `{ username, password, phone?, dingtalk_id? }` |
| 响应 | `{ access_token, username }` |

实际响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "access_token": "jwt-token",
    "username": "zhangsan"
  }
}
```

前端适配说明：

| 前端原设计 | 实际接口情况 | 调整建议 |
|---|---|---|
| 登录时传 `role` | 实际登录接口不接收 `role` | 登录表单中的角色选择可暂时移除，或仅作为前端演示字段不提交 |
| 登录后调用 `/api/auth/me` 获取菜单 | `/api/auth/me` 返回用户信息和 role，但不返回 menus | 前端根据 role 本地生成默认菜单，或后端补充 menus 字段 |
| 登录返回 `token_type` | 实际未返回 | 前端固定使用 `Bearer ${access_token}` |

#### 5.1.2 当前用户

| 项目 | 内容 |
|---|---|
| 方法 | GET |
| 实际路径 | `/api/auth/me` |
| Header | `Authorization: Bearer <token>` |
| 响应字段 | `user_id`、`username`、`role`、`phone`、`dingtalk_id` |

前端用户 Store 建议结构：

```ts
interface CurrentUser {
  user_id: number
  username: string
  role: string
  phone?: string | null
  dingtalk_id?: string | null
  menus?: MenuItem[]
}
```

`menus` 当前由前端按角色生成。

#### 5.1.3 退出登录

| 项目 | 内容 |
|---|---|
| 方法 | POST |
| 实际路径 | `/api/auth/logout` |
| 说明 | 后端返回成功提示，前端仍需清理本地 Token、Pinia Store 和路由状态 |

#### 5.2.5 角色管理

当前实际接口只提供角色基础 CRUD 和 `permissions` 数组字段，没有独立权限树接口。

| 页面需求 | 当前实际接口 | 处理建议 |
|---|---|---|
| 角色列表 | `/api/roles/list` | 可直接联调 |
| 新增角色 | `/api/roles/add` | 可直接联调 |
| 编辑角色权限树 | `/api/roles/{id}` + `/api/roles/update` | 前端将权限树选中的 key 转为 `permissions: string[]` |
| 删除角色 | `/api/roles/{id}` | 可直接联调 |
| 获取权限树模板 | 未提供 | 前端本地维护权限树模板，或后端新增 `/api/v1/permissions/tree` |

建议前端本地权限模板：

```ts
const permissionTree = [
  {
    id: 'dashboard',
    name: '监测看板',
    children: [
      { id: 'dashboard:view', name: '查看' },
      { id: 'dashboard:edit', name: '编辑' },
      { id: 'dashboard:delete', name: '删除' }
    ]
  }
]
```

保存时调用：

```http
PUT /api/roles/update
```

请求体：

```json
{
  "id": 1,
  "permissions": ["dashboard:view", "alerts:view"]
}
```

#### 6.2.1 实时数据 Tab

- COD
- 氨氮
- 总磷
- 总氮
- 水温

const service = axios.create({

service.interceptors.request.use(config => {

  }

### 15.2 统一响应拦截

```ts
service.interceptors.response.use(
  response => {
    const res = response.data
    if (res.code !== 0) {
      ElMessage.error(res.message || '请求失败')
      return Promise.reject(res)
    }
    return res.data
  },
  error => {
    ElMessage.error('网络请求失败，请检查网络连接')
    return Promise.reject(error)
  }
)
```

### 15.3 分页适配函数

```ts
export function normalizePage<T>(data: {
  lists: T[]
  pagination: {
    page: number
    page_size: number
    total: number
    total_pages: number
  }
}) {
  return {
    items: data.lists || [],
    page: data.pagination?.page || 1,
    pageSize: data.pagination?.page_size || 10,
    total: data.pagination?.total || 0,
    totalPages: data.pagination?.total_pages || 0
  }
}
```

## 20. 最终结论

本版本已经将上一版“前端页面驱动设计”调整为“实际 API 风格对齐设计”。

当前实际 API 已经可以支撑以下页面的基础联调：

1. 登录页；
2. 首页告警总览的统计与水库卡片；
3. 水库实时监测详情的基础信息、站点列表、趋势查询；
4. 预警中心列表；
5. 预警详情基础信息；
6. 水库、站点、指标配置管理；
7. 用户与角色基础管理。

仍需补齐的核心能力是：

1. 水库详情九项最新指标聚合；
2. 预警状态流转与处置备注；
3. 预警溯源图谱；
4. RAG 智能问答；
5. 知识库管理；
6. 报告中心；
7. 水源调度建议；
8. WebSocket / SSE 实时推送。

前端开发时应优先对接“已实现接口总览”中的实际路径，并在请求封装层完成 `lists + pagination`、枚举值、字段名称和 Token 认证的统一适配。后端新增接口时建议统一采用 `/api/v1` 前缀，并继续沿用当前 `ApiResponse` 与分页结构。
